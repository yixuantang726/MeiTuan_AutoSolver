# AutoSolver：多策略擂台赛框架下的快递派单优化系统

## 摘要

本文提出AutoSolver，一个融合多种组合优化算法与大语言模型(LLM)裁判机制的快递派单优化系统。系统采用**擂台赛竞争框架**，让贪心、启发式、整数规划等五种求解策略进行轮次竞争，由J-Agent充当独立裁判，而非平行求解策略。这样设计的核心优势在于：避免将LLM的推理过程作为优化求解的直接参与者，从而保证了求解结果的可解释性和可验证性。同时，通过λ参数的自适应调节，系统灵活控制覆盖率惩罚的力度，实现在成本与服务质量间的动态平衡。

## 1 问题定义与优化目标

### 1.1 快递派单问题

给定：
- **40个配送任务集合** $T = \{T_0001, T_0002, \ldots, T_{0040}\}$
- **80个骑手的派单候选** $P = \{p_1, p_2, \ldots, p_{80}\}$，每条候选 $p_i$ 包含：
  - `task_key`: 包含的任务列表（支持合单，多个任务集合）
  - `courier_id`: 骑手ID
  - `score`: 派单成本（权重）
  - `willingness`: 接单概率（0-1）

求：从候选池中选择子集 $S \subseteq P$，最小化：

$$F(S) = R(S) + \lambda \times (40 - C(S))$$

其中：
- **期望成本** $R(S) = \sum_{p_i \in S} \text{score}_i \times \text{willingness}_i$
- **覆盖率** $C(S) = \sum_{t \in T} \left(1 - \prod_{p_j: t \in p_j, p_j \in S} (1-\text{willingness}_j)\right)$
  - 支持**冗余派发**：同一任务可被多条候选覆盖，成功接单概率为 $1 - \prod(1-w)$
- **λ参数**：覆盖率惩罚系数，体现"未接单的代价"

### 1.2 λ的启发式本质

λ的设置体现了一种启发式规则：当覆盖某个额外任务的边际成本为 $\Delta R$ 时，若 $\lambda > \Delta R$，系统会倾向于增加派单覆盖；反之则优先控制成本。因此：
- **λ=30（初始）**：覆盖一个任务的"惩罚"为30元，激励覆盖
- **λ↑（连败3轮后，S擂主赢）**：说明现有方案覆盖不足，需增强覆盖驱动
- **λ↓（连败3轮后，C挑战者赢）**：说明现有方案追求覆盖过度，需强调成本控制

这是**显式的启发式权衡**，而非黑箱的LLM推理。

### 2.2 Agent角色定义

| Agent | 角色 | 职责 | 调用对象 |
|-------|------|------|---------|
| **C-Agent** | 覆盖优先方 | 每轮调用 `run_coverage_greedy` 获方案 | 贪心工具 |
| **S-Agent** | 成本优先方 | 每轮调用 `run_score_greedy` 获方案 | 贪心工具 |
| **J-Agent** | 唯一裁判 | 不参与求解，仅调用 `compare_solutions` 比较F值，拥有唯一"TERMINATE"权限 | 评估工具 |
| **M-Agent** | 资源规划者 | 仅在J-Agent召唤时介入，查找合单机会 | 合单工具 |

**关键设计**：J-Agent 严格分离为**决策者而非求解者**。它不产生新的派单方案，仅基于工具提供的数值结果做出透明的比较判断，避免LLM的推理偏差对优化结果的黑箱影响。

## 3 擂台赛竞争框架

### 3.1 状态机流程

```
初始化（λ=30）
    ↓
[第0轮] Greedy基线（无LLM）
  C_coverage_greedy F=f₁ vs S_score_greedy F=f₂
  → 确定初始擂主（F值较低者），记录到历史
    ↓
[第1-10轮] 擂主保卫战
  ① 擂主：直接复用上轮赢时的锁定解（不重计算）
  ② 挑战者：穷尽所有策略{greedy, heuristic, ILP, bundle_heuristic}，选最优F值
  ③ J-Agent：
     - 调 compare_solutions(lam=λ) 获取两方案的完整指标对比
     - 比较 F 值：较低者胜（含覆盖率惩罚，无需单独判断）
     - 检查胜者未覆盖T，若有 + 骑手还有空闲 → 说"M_Agent请介入"
     - 否则直接"TERMINATE"（结束本轮，进入下一轮）
  ④ 如J-Agent说"M_Agent请介入"：
     - M-Agent调 find_winner_bundle_options，寻合单方案
     - 推荐 ROI 最高者（覆盖未覆盖T数/成本），说一句理由
     - 继续"TERMINATE"
  ↓
[胜者判定]
  - 挑战者F < 擂主F：挑战者升为新擂主，原擂主为新挑战者
    新挑战者连败次数清零，从非greedy策略重新开始
  - 否则：擂主保位，连败次数+1
  ↓
[λ调节]
  if 连败次数 == 3:
    if 擂主==S: λ *= 1.3 (max=100)  // 覆盖不足，提升覆盖驱动
    else:      λ *= 0.8 (min=10)   // 覆盖过度，强调成本控制
    重置连败计数
```

### 3.2 核心创新：擂主锁定与挑战者全策略搜索

- **擂主锁定**：赢方的派单方案被锁定，下轮直接复用其F(S)与覆盖率，无需重新运算。这减少了计算开销，同时确保稳定性。
- **挑战者全策略搜索**：每轮挑战者不仅运行贪心，更尝试启发式、ILP、合单等高阶策略，选F值最低者发起挑战。这激励算法创新。
- **λ自适应**：连败3轮自动调节λ（范围[10, 100]），平衡覆盖与成本，无需人工干预。

## 4 J-Agent设计：为何不将LLM作为求解策略

### 4.1 设计决策

在初期设计中，我们考虑过让LLM直接生成派单方案（第五种策略），但最终选择：
- **LLM专职为裁判**，不作为求解策略参与者

### 4.2 核心理由

**1. 推理证据的透明性** 
- 贪心、启发式、ILP都基于精确的数学定义和明确的目标函数。用户可追溯为何选择某个方案。
- LLM推理通常是黑箱：其决策依据难以被解释，生成的派单方案缺乏数学正当性。

**2. 优化结果的可验证性**
- 若LLM方案胜出，我们无法回答"为什么这个方案更好"的根本问题。只能说"LLM说它好"。
- J-Agent使用工具计算的F(S)可精确复现，便于审计和故障排查。

**3. 避免评价标准的混乱**
- 在同一擂台赛中混用LLM的"推理质量"和算法的"F值"，等同于混用两种度量。
- J-Agent采用统一的F值标准裁判，确保所有候选方案在同一标准线上竞争。

**4. 实际可部署性**
- LLM推理有延迟、成本、服务依赖。
- 数学算法可离线运行，稳定可靠，易于集成到生产环境。

### 4.3 J-Agent的工具化实现

```python
# J-Agent系统提示词的核心
def j_agent_system_message():
    return """你是J-Agent，本轮唯一的裁判。
    
    【职责1：比较两方案】
    1. 调用 compare_solutions(lam=λ) 工具
    2. 获取C和S两个方案的F(S), coverage, expected_score
    3. 比较标准：F(S)越低越好
    4. 宣布较优方案，格式：「X_Agent胜」
    
    【职责2：判断补单机会】
    1. 调用 check_winner_uncovered(lam=λ) 检查未覆盖T
    2. 若 worth_patching=True，说「M_Agent请介入」
    3. 否则直接「TERMINATE」
    
    注意：你是唯一能说TERMINATE的Agent。
    所有判断都通过工具完成，不要凭空猜测数字。"""
```

所有决策基于工具调用的结果，而非LLM的自由推理。

## 5 λ参数的启发式设计

### 5.1 覆盖率惩罚的含义

λ 不仅是一个技术参数，更是**业务规则的数学表达**：

- **λ=30**：每未覆盖一个任务，相当于"欠债30元"。只有当派单成本 < 30元时，才值得为了覆盖这个任务而派单。
- **λ=39**（经过调节）：加强覆盖驱动，表示上一阶段"成本控制过度，必须覆盖更多"。
- **λ=24**（经过调节）：削弱覆盖驱动，表示"合单与覆盖已经充分，进一步增加会不划算"。

### 5.2 自适应调节逻辑

```python
if 连败3轮:
    if 擂主 == "S"（成本优先方一直赢）:
        λ *= 1.3  # S赢意味着成本控制很好，但覆盖可能不足
              # 增强覆盖惩罚，让下一轮更倾向选择覆盖更多的方案
    else （C挑战者一直赢）:
        λ *= 0.8  # C赢意味着覆盖优先，但成本急剧上升
              # 削弱覆盖惩罚，让下一轮更重视成本控制
```

这是一种**显式的启发式反馈机制**，比任何隐式的LLM权衡更透明、更可调。

## 6 评估函数与冗余派发

### 6.1 覆盖率计算的核心：冗余派发支持

当同一任务被多个骑手覆盖时，实际接单概率为：

$$P(\text{任务被接}) = 1 - \prod_{i: \text{骑手}i\text{覆盖该任务}} (1 - w_i)$$

这反映了现实中"有人接单就行"的逻辑，而非"需要所有人都接"。

### 6.2 期望成本的顺序模型

对于每个任务 $t$，若被骑手 $i_1, i_2, \ldots, i_k$ 覆盖（按选择顺序），期望成本为：

$$E[\text{cost}_t] = \sum_{j=1}^{k} s_{i_j} \cdot w_{i_j} \cdot \prod_{m<j}(1-w_{i_m})$$

即：第 $j$ 个骑手接单的概率 = 前面所有人都拒绝 × 自己接受。

这保证了无冗余派发方案（如greedy/ILP输出）的计算结果与原始 evaluate() 函数完全一致。

## 8 总结与展望

AutoSolver的核心贡献在于：

1. **清晰的问题建模**：快递派单优化的精确数学表达，包括冗余派发和概率模型。

2. **透明的多策略竞争框架**：让多个确定性算法在同一擂台赛中竞争，而非混杂黑箱LLM推理。

3. **工具化的LLM决策**：J-Agent严格充当"数值比较器"角色，避免推理证据不透明对优化结果的污染。

4. **显式的启发式平衡**：λ的自适应调节直观体现覆盖与成本的权衡，易理解、易调参、易扩展。

5. **全面的可追溯性**：每一步决策都有数学根据，整个过程可验证、可审计、可改进。

这样的设计使得系统既能利用LLM的决策能力，又不牺牲数学优化的严谨性和可解释性。在快递物流、任务调度等实际应用中，这种"LLM as Judge, Algorithm as Solver"的范式具有显著的生产价值。
